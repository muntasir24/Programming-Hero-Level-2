# 📦 Modular System and IIFE: Solving the Global Scope Problem

When writing JavaScript, managing variables across multiple files can quickly become a disaster. Let's explore how JavaScript evolved from messy global variables to **IIFE**, and finally to the modern **Modular System**.

---

## Step 1: What is an IIFE (Immediately Invoked Function Expression)?

```mermaid
flowchart TD
    subgraph Danger Zone (Global Scope)
    File1[File 1: let a = 10]
    File2[File 2: let a = 20]
    File1 -- "Variable 'a' Collides! 💥" --> File2
    end
    
    subgraph Safe Zone (Functional Scope / IIFE)
    I1[(IIFE 1: let a = 10)]
    I2[(IIFE 2: let a = 20)]
    I1 -. "Safe, completely hidden! 🛡️" .- I2
    end
```

### A. What it is
An **IIFE** (Immediately Invoked Function Expression) is a JavaScript function that runs the exact millisecond it is defined. Its primary superpower is that it takes variables out of the public "Global Scope" and traps them safely inside a private "Functional Scope". 

### B. The Problem (Global Scope Pollution)
Before modern JavaScript, if you linked two different JavaScript files in your HTML, they shared the exact same public space (Global Scope). If Developer A created a variable named `a` or a function named `add` in File 1, and Developer B accidentally used the exact same names in File 2, the app would crash or behave weirdly because the names collided.

**Problem Code ( Scope Collision):**
```javascript
// ❌ First File (Global Scope)
function f1() {
    console.log("Korim"); // Developer 1 wants this to print "Korim"
}

// ❌ Second File (Global Scope)
// Developer 2 unknowingly uses the exact same function name!
function f1() {
    console.log("Rahim"); 
}

// 🛑 CRASH/COLLISION! The Global Scope is polluted.
// The second function permanently overwrites the first one!
f1(); // Output: "Rahim" (Korim is destroyed forever!)
```

### C. The Solution (Creating Functional Scope via IIFE)
To solve this, developers wrapped their entire file's code inside an anonymous function and instantly invoked it `()`. Because variables and functions declared inside a function cannot be seen from the outside, the names are no longer in the Global Scope. They are safely hidden in the **Functional Scope**.

**Solution Code:**
```javascript
// ✅ First File (Wrapped in IIFE)
(function f1() {
    // 1. This function is trapped safely inside its own execution block!
    console.log("Korim");
})(); // Outputs: "Korim"

// ✅ Second File (Wrapped in IIFE)
(function f1() {
    // 2. This function is ALSO trapped. 
    // No collision! They don't know each other exists and both print perfectly.
    console.log("Rahim");
})(); // Outputs: "Rahim"
```

### D. Real-Life Analogy
💡 **Two Meetings in the Same Hall vs Soundproof Rooms.**
*   **Global Scope (The Problem):** Two different business teams are trying to hold their meetings in the middle of a massive open stadium. People are shouting, talking over each other, and confusing their presentations.
*   **IIFE (The Solution):** You put each team inside their own **Soundproof Glass Room (Functional Scope)**. Both rooms can have a guy named "John" (variable `a`), and they will never confuse each other because the sound doesn't leave the room.

**Analogy Code:**
```typescript
// ❌ Open Stadium (Collision)
let meetingTopic = "Sales"; 
let meetingTopic = "Engineering"; // Crash! Cannot have two!

// ✅ Soundproof Rooms (IIFE)
(() => {
    let meetingTopic = "Sales Strategy"; // Safe inside Room A
})();

(() => {
    let meetingTopic = "Engineering Architecture"; // Safe inside Room B
})();
```

## Step 2: Global Scope vs Functional Scope

```mermaid
flowchart TD
    G((🌍 Global Scope: The Public Square))
    F1((🔒 Functional Scope A: Private Room))
    F2((🔒 Functional Scope B: Private Room))
    
    G -->|Visible to Everyone| F1
    G -->|Visible to Everyone| F2
    F1 -.x|100% Invisible| G
    F2 -.x|100% Invisible| G
    F1 -.x|100% Invisible| F2
```

### A. What it is
*   **Global Scope:** Any variable written openly in a file (outside of any function) lives in the Global Scope. These variables can be seen, read, and changed by *any* other file or function in the entire program.
*   **Functional Scope:** Any variable created inside a function (e.g., using `let` or `const` inside an IIFE) lives in the Functional Scope. These variables are strictly private to that function and absolutely invisible to the outside world.

### B. The Problem (The Global Free-For-All)
If you rely on the Global Scope, ANY piece of code can accidentally overwrite your important data. 

**Problem Code:**
```javascript
// ❌ Global Scope (Too Public!)
let myBankBalance = 5000; // Anyone can touch this!

function hackerPlugin() {
    // A completely unrelated function can manipulate the global variable!
    myBankBalance = 0; 
}

hackerPlugin();
console.log("My money is gone!", myBankBalance); // Output: 0
```

### C. The Solution (Protection via Functional Scope)
By placing sensitive variables inside a Functional Scope, you create an impenetrable wall. Even if an outside function tries to modify the data, it literally cannot see it.

**Solution Code:**
```javascript
// ✅ Functional Scope (Protected!)
(() => {
    // This variable only exists inside this block!
    let secureBankBalance = 5000; 
    console.log("Safe inside:", secureBankBalance);
})();

function hackerPlugin() {
    // 🛑 ReferenceError! The hacker function cannot see into the Functional Scope!
    secureBankBalance = 0; 
}
```

### D. Real-Life Analogy
💡 **The Public Park vs Your Bedroom.**
*   **Global Scope (Public Park):** You leave your perfectly drawn painting on a bench in the park. Anyone walking by can accidentally spill coffee on it, draw a mustache over it, or steal it.
*   **Functional Scope (Your Bedroom):** You draw the painting and immediately lock it inside your bedroom. The people in the public park (Global Scope) don't even know the painting exists.

**Analogy Code:**
```typescript
class PublicPark {
    publicBenchItem = "My Masterpiece Painting"; // Global
}

class BedroomScope {
    private hiddenItem = "My Masterpiece Painting"; // Functional
    
    viewItem() { return this.hiddenItem; }
}
```

## Step 3: Where is IIFE Used in Real-Life Software Development?

### A. What it is
Even though modern frameworks use ESM (Modules), IIFE is strictly NOT dead. Professional developers still use IIFEs today to solve very specific problems where a block of code needs isolated execution without leaving a permanent footprint in the surrounding scope.

### B. The Problem (Dirtying the Global File or Hook)
There are two major modern scenarios where NOT using an IIFE causes pain:
1. **Top-Level Await Limitation:** In older JS or specific configs, you cannot just drop an `await` command out in the open root of a file. It causes syntax errors.
2. **React `useEffect` Rule:** React absolutely forbids the `useEffect` hook callback from being an asynchronous function directly. If you try to write `useEffect(async () => {...})`, React crashes.

**Problem Code:**
```javascript
// ❌ Using await in the open causes problems in older environments
await fetchDatabase(); // Error: await is only valid passing an async function!

// ❌ React restricts async directly inside useEffect
import { useEffect } from 'react';
useEffect(async () => {
    // 🛑 React crashes! useEffect must return a cleanup function, not a Promise!
    const data = await fetchUsers(); 
}, []);
```

### C. The Solution (Using IIFE in the Real World)
**1. Async IIFE for Top-Level Execution:** We wrap the `await` in an asynchronous IIFE so it can run immediately without polluting the rest of the file.
**2. IIFE inside React `useEffect`:** We can trick React by creating an isolated async function inside the hook and immediately invoking it (IIFE). This perfectly obeys React's strict rules while still giving us async power.

**Solution Code:**
```javascript
// ✅ 1. Real-Life Top-Level Async IIFE (Safe execution)
(async () => {
    try {
        console.log("Starting DB Connection...");
        const db = await fetchDatabase(); // Perfectly safe inside the IIFE!
        console.log("Connected!");
    } catch(err) {
        console.log(err);
    }
})(); // Immediately Invoked!

// ✅ 2. Real-Life React useEffect Async Fetching
import { useEffect, useState } from 'react';

function UserList() {
    const [users, setUsers] = useState([]);
    
    useEffect(() => {
        // We use an Async IIFE here! React is completely fine with this.
        (async () => {
            const data = await fetch('api/users');
            const json = await data.json();
            setUsers(json);
        })(); // Invoked immediately inside the hook
    }, []);
    
    return <div>Loaded Users Safely!</div>;
}
```

### D. Real-Life Analogy
💡 **A Pop-up Food Stall (The Real-World IIFE).**
* **The Permanent Restaurant (Standard Function):** You buy a huge plot of land, build a physical restaurant, register a permanent name, and someone must actively walk in to order food.
* **The Pop-Up Stall (IIFE):** You set up a tiny tent at a festival, instantly serve 500 burgers right then and there, and the moment the festival is over, the tent completely vanishes like it never existed. It did its job immediately and left zero footprint/mess behind for the city to clean up.

**Analogy Code:**
```typescript
// Permanent Restaurant (Standard)
function permanentDiner() {
    console.log("I exist forever. Call me anytime.");
}
permanentDiner(); // Need to call it later

// Pop-up Stall (IIFE)
(() => {
    console.log("🍔 I popped up! Served burgers! Now I vanish from memory safely!");
})(); // Fired immediately, leaves no permanent structure!
```

## Step 4: Why the Modular System Came (CommonJS vs ESM)

```mermaid
flowchart LR
    A[math.js] -->|Explicitly Exports| B((The Bridge))
    B -->|Explicitly Imports| C[app.js]
    
    subgraph CommonJS (Node.js Server)
    D[require]
    end
    
    subgraph ESM (Client / Modern JS)
    E[import / export]
    end
```

### A. What it is
A **Modular System** allows developers to split up their code into separate, independent files (called Modules) where everything is completely private by default. Unlike an IIFE, a Module allows you to *intentionally* share (export) specific functions and grab (import) them exactly where you need them.

### B. The Problem (IIFEs can't share easily)
IIFEs were great for hiding code, but what if File 2 *needed* to use the `add` function from File 1? Because the IIFE made it 100% private and soundproof, File 2 couldn't access it! Early developers had to use ugly hacks, like purposefully leaking variables to `window.add`, which recreated the exact same mess they were trying to avoid. Furthermore, as Node.js grew, managing hundreds of files manually was impossible.

**Problem Code:**
```javascript
// ❌ IIFE hides things TOO well, making sharing difficult
(() => {
    let add = (a, b) => a + b;
    // Hack: Forcing it back into the global scope just to share it!
    window.globalAdd = add; 
})();

(() => {
    // 🛑 We have to rely on ugly global variables again to share data.
    console.log(window.globalAdd(5, 10)); 
})();
```

### C. The Solution (Modules)
Node.js introduced **CommonJS**, a beautiful system where every file is naturally isolated (no IIFE needed!). You simply use `module.exports` to share only what you want, and `require()` to bring it into another file. Later, modern JavaScript introduced **ESM (ECMAScript Modules)** which does the same thing using `export` and `import`.

**Solution Code:**
```javascript
// ✅ 1. CommonJS System (Heavily used in Node.js Server side)
// File: server.js
const express = require("express"); 
const cors = require("cors");
const jwt = require("jsonwebtoken");

console.log("CommonJS successfully imported modules secretly and cleanly!");


// =========================================================================


// ✅ 2. ESM (ES6) Module System (Heavily used in React / Client side)
// File: App.jsx
import React, { useState } from "react";
import axios from "axios";
import { useDispatch, useSelector } from "react-redux";

console.log("ESM completely replaced the need for ugly IIFE hacks!");
```

### D. Real-Life Analogy
💡 **Lego Blocks.**
*   **IIFE System:** Imagine taking all your Lego pieces and permanently supergluing them together into one giant unchangeable ball so nobody loses them. They are safe, but you can't reuse the wheels for another car.
*   **Modular System:** Every Lego piece (Module) comes in its own little box. By default, they are separate. But each piece has standardized "bumps" (Exports) and "holes" (Imports). You can snap the `express` wheel block perfectly into your `server.js` car block exactly when and where you need it!

**Analogy Code:**
```typescript
// 📦 Block 1: mathModule.ts
class MathLegoBlock {
    // We intentionally EXPORT the bump
    export_add(a: number, b: number) { return a + b; } 
}

// 📦 Block 2: app.ts
class AppLegoBlock {
    // We strictly IMPORT the hole
    import_run() {
        const math = new MathLegoBlock();
        console.log(`Snapping blocks together! Total: ${math.export_add(10, 20)}`);
    }
}
```

## Step 8: Node.js and ESM (The `.mjs` Extension)

```mermaid
flowchart LR
    subgraph file1.mjs (Sender)
    E[export default a]
    end

    subgraph file2.mjs (Receiver)
    I[import a from './file1.mjs']
    end

    E -- "Modern ES Modules 🚀" --> I
```

### A. What it is
By default, Node.js uses the older CommonJS (`require` / `module.exports`) system. If you want to use the modern **ESM (ECMAScript Modules)** syntax (`import` / `export`) without modifying the entire project's `package.json`, you can simply change the file extension from `.js` to **`.mjs`** (Modular JavaScript).

### B. The Problem (Node.js Does Not Understand Default `import`)
If you write modern `import` and `export` inside a standard `.js` file, Node.js will crash immediately. It expects `require()` and gets confused by the modern syntax. Furthermore, if you forget to export a variable from `file1`, `file2` cannot use it.

**Problem Code:**
```javascript
// ❌ file1.js (Standard JS file)
const a = 1000;
// We forgot to export... Or we used 'export default' in a normal .js without config
export default a; // 🛑 Node.js throws SyntaxError: Cannot use import statement outside a module

// ❌ file2.js 
import a from "./file1.js"; // 🛑 Crash!
```

### C. The Solution (`.mjs` Files)
By renaming our files to `.mjs`, we explicitly tell Node.js: "Hey! Treat this file as a modern ECMAScript Module!" Now, we can use `export default` to send a single variable (`a = 1000`) out, and cleanly `import` it into another file. Note: With ESM, you MUST include the file extension (`.mjs`) inside the import path!

**Solution Code:**
```javascript
// ✅ file1.mjs (The Sender)
const a = 1000;

// Perfectly valid in an .mjs file!
export default a; 


// =========================================================================


// ✅ file2.mjs (The Receiver)
// Must include the exact '.mjs' extension in the path!
import a from "./file1.mjs";

console.log(a); // Output: 1000
```

### D. Real-Life Analogy
💡 **The Old Post Office vs The Modern Courier.**
*   **The Problem:** You try to hand a modern digital package (`import/export`) to a 100-year-old Post Office (`.js`). The worker looks at it, gets completely confused, and rejects your package because they only accept old handwritten letters (`require`).
*   **The Solution (`.mjs`):** You put a bright modern sticker on the box (`.mjs`). Now, a modern fast courier service instantly recognizes it and smoothly delivers it to the next house. 

**Analogy Code:**
```typescript
// 📦 Old System (.js)
class OldPostOffice {
    processLetter() { return "Using require() only!"; }
}

// 🚀 Modern Courier (.mjs)
class ModernCourier {
    // Explicitly labeled as modern `.mjs` module
    export_deliver() {
        const a = "📦 High-Speed Package"; 
        return a;
    }
}

// 🏠 Your House (file2.mjs)
class MyHouse {
    import_receive() {
        const packageReceived = new ModernCourier().export_deliver();
        console.log(`Successfully received: ${packageReceived}`);
    }
}
```

## Step 5: CommonJS in Large Projects (Destructuring & Aliasing)

```mermaid
flowchart TD
    F1[file1.js: export { a: 100 }]
    F3[file3.js: export { a: 200 }]
    F2((file2.js))
    
    F1 -- "Import 'a' as 'x'" --> F2
    F3 -- "Import 'a' as 'y'" --> F2
    F2 -- "console.log(x, y)" --> Out[Printed: 100 200]
```

### A. What it is
In massive Node.js projects, we use the **CommonJS** module system (`require` and `module.exports`) instead of IIFEs. Sometimes, you want to export an object containing multiple variables (`module.exports = { a }`), and use **Destructuring** (`const { a } = require(...)`) to pull exactly what you need. But when two different files export a variable with the exact same name, we use **Aliasing** (`a : x`) to rename them upon import to prevent crashing.

### B. The Problem (Naming Collisions During Import)
If we have a big website with hundreds of modules, two different files (`file1.js` and `file3.js`) might both export a variable named `a`. If we try to import them both into `file2.js` using regular destructuring, JavaScript will crash because we are trying to declare the variable `a` twice in the same file!

**Problem Code:**
```javascript
// ❌ file1.js
const a = 100;
module.exports = { a };

// ❌ file3.js
const a = 200;
module.exports = { a };

// ❌ file2.js (The Crash!)
const { a } = require('./file1');
// 🛑 SyntaxError: Identifier 'a' has already been declared!
const { a } = require('./file3'); 
```

### C. The Solution (CommonJS Aliasing)
We solve this by renaming the variables *at the exact moment* we destructure them. We use the syntax `const { originalName : newName } = require(...)`. This allows us to secretly rename `a` from `file1` to `x`, and `a` from `file3` to `y`. No more collisions!

**Solution Code:**
```javascript
// ✅ file1.js
const a = 100;
module.exports = { a };

// ✅ file3.js
const a = 200;
module.exports = { a };

// ✅ file2.js (Handling Collisions flawlessly!)
// We take 'a' from file1 but rename it to 'x'
const { a : x } = require('./file1');

// We take 'a' from file3 but rename it to 'y'
const { a : y } = require('./file3');

console.log(x, y); // Output: 100 200 (Collision safely resolved!)
```

### D. Real-Life Analogy
💡 **Two Employees with the Same Name.**
*   **The Problem:** You have two employees named "John" transferring to your department from two different branches (Branch 1 and Branch 3). If you just yell "Hey John!" in the office, both will answer, causing massive confusion.
*   **The Solution (Aliasing):** The moment they walk through the door, you assign them nicknames (Aliases). You say, "John from Branch 1 will now be known as `Mr. X`" and "John from Branch 3 will now be known as `Mr. Y`." Now you can talk to both securely!

**Analogy Code:**
```typescript
class BranchOne {
    public employee = "John";
}
class BranchThree {
    public employee = "John";
}

// 🏢 Main Office (Importing both Johns)
const branch1 = new BranchOne();
const branchThree = new BranchThree();

// ✅ Aliasing to prevent confusion!
const { employee: mrX } = branch1;
const { employee: mrY } = branchThree;

console.log(`Welcome ${mrX} and ${mrY}! No name collisions here!`);
```

## Step 6: Reusing Functions via Utility Modules

```mermaid
flowchart TD
    subgraph Utility Module
    U[utils.js: Utility Box]
    U -->|export { add }| Add[add()]
    U -->|export { subtract }| Sub[subtract()]
    end

    subgraph Application
    A[app.js: Main File]
    A -.->|const { add } = require| Add
    A -.->|const { subtract } = require| Sub
```

### A. What it is
A **Utility Module** (often named `utils.js` or `helpers.js`) is a dedicated file used purely to store helper functions—like mathematical calculations, date formatting, or string manipulations. Instead of making functions restricted and hidden inside IIFEs, we explicitly export these functions so multiple files across the entire application can share and reuse them endlessly.

### B. The Problem (Code Duplication Nightmare)
If you do not use modular utility files, you will end up writing the exact same helper function in `file1.js`, `file2.js`, and `file3.js`. If you ever need to fix a bug in that function, you have to manually find and update it in every single file, which violates the **DRY (Don't Repeat Yourself)** principle.

**Problem Code:**
```javascript
// ❌ file1.js (Checkout Page)
const calculateTax = (amount) => amount * 0.15;
console.log("Electronics Tax:", calculateTax(1000));

// ❌ file2.js (Admin Dashboard)
// Disaster: Duplicating the exact same function everywhere!
const calculateTax = (amount) => amount * 0.15; 
console.log("Clothing Tax:", calculateTax(500));
```

### C. The Solution (The Utility File)
We extract the reusable function into its own `utils.js` file, export it using `module.exports`, and then `require()` it (or `import` it) anywhere we need it. This way, the logic exists in only **one single place**, keeping the codebase incredibly clean.

**Solution Code:**
```javascript
// ✅ 1. The Utility File (utils.js)
const calculateTax = (amount) => amount * 0.15;
const formatCurrency = (amount) => `$${amount.toFixed(2)}`;

// Exporting multiple utility functions!
module.exports = { calculateTax, formatCurrency };


// =========================================================================


// ✅ 2. The Application File (app.js)
// Destructuring exactly the functions we need from the utility file
const { calculateTax, formatCurrency } = require('./utils');

const tax = calculateTax(1000);
console.log("Tax to pay:", formatCurrency(tax)); // Output: Tax to pay: $150.00
```

### D. Real-Life Analogy
💡 **The Community Tool Shed.**
*   **The Problem:** You buy a hammer for your kitchen, buy a second hammer for your garage, and buy a third hammer for your bedroom. It's a massive waste of money, and if a hammer breaks, you have to replace three of them.
*   **The Solution (Utility File):** You build a **Community Tool Shed (utils.js)** and put exactly ONE hammer inside it. Now, whenever the kitchen or the garage needs a hammer, they just import it from the shed, use it, and you only ever have to maintain one tool.

**Analogy Code:**
```typescript
// 🛠️ The Utility File (ToolShed)
class ToolShed {
    // Utility functions shared with everyone
    static getHammer() {
        return "🔨 Hammer";
    }
    static getScrewdriver() {
        return "🪛 Screwdriver";
    }
}

// 🏠 The Application (Kitchen)
class Kitchen {
    repairCabinet() {
        // We just borrow the utility function! No need to build our own hammer!
        const tool = ToolShed.getHammer();
        console.log(`Fixing cabinet safely with ${tool}!`);
    }
}

const myKitchen = new Kitchen();
myKitchen.repairCabinet();
```

## Step 7: How `index.js` Solves Everything (The Barrel Pattern)

```mermaid
flowchart TD
    subgraph The utility Folder
    A[add.js<br>f1()] -->|Exports to| IDX((index.js<br>The Central Hub))
    S[sub.js<br>f2()] -->|Exports to| IDX
    end

    IDX -->|Single clean import| F2[file2.js]
```

### A. What it is
In Node.js, `index.js` is a magic file. When you import a folder (e.g., `require('./utility')`), Node.js automatically looks for a file named `index.js` inside that folder. We use `index.js` as a **Barrel File** or a **Central Hub** that collects exports from many different small files (`add.js`, `sub.js`), bundles them together, and exports them as one single package.

### B. The Problem (Importing a Million Files)
As your project grows, your `utility` folder might have many files. If your main application needs functions from all of them, your file will be cluttered with a massive, ugly block of `require` statements targeting each exact file.

**Problem Code:**
```javascript
// ❌ file2.js (The Ugly Import Mess)
// We have to specifically target every single file inside the utility folder!
const add = require('./utility/add.js');
const sub = require('./utility/sub.js');

console.log("Additions : ", add(100, 200));
```

### C. The Solution (The `index.js` Hub + Aliasing)
We create a new file named `index.js` right inside the `utility/` folder. Its only job is to import everything from its sibling files (`add.js`, `sub.js`) and export them together as `{f1, f2}`. In our `file2.js`, we do ONE clean import from the `utility/` folder. We also use **Aliasing** (`f1 : add`) to nicely rename them back to meaningful names!

**Solution Code:**
```javascript
// ✅ 1. Inside utility/add.js
function f1(a, b) {
    const add = a + b;
    return add;
}
module.exports = f1;

// ✅ 2. Inside utility/index.js (The Central Hub)
const f1 = require('./add');
const f2 = require('./sub');

// Bundle them up into one object!
module.exports = { f1, f2 };

// =========================================================================

// ✅ 3. Inside file2.js (The Consumer)
const { a: x } = require('./file1'); // 100
const { a: y } = require('./file3'); // 200

// We do ONE clean import from the folder itself.
// We use ALIASING to rename 'f1' to 'add' and 'f2' to 'sub'
const { f1: add, f2: sub } = require('./utility'); 

console.log("Additions : ", add(x, y)); // Outputs: Additions : 300
```

### D. Real-Life Analogy
💡 **Going to the Supermarket vs Going to Individual Farms.**
*   **The Problem:** You want to bake a cake. Oh no! You have to drive to the Dairy Farm (`add.js`) for milk and drive to the Chicken Farm (`sub.js`) for eggs. It takes forever.
*   **The Solution (`index.js`):** You just drive to ONE **Supermarket (`index.js`)**. The supermarket already did the hard work of gathering the milk (`f1`) and eggs (`f2`) from all the separate farms and put them in one building. You just walk in, grab exactly what you need in one trip, name them properly in your kitchen, and go home!

**Analogy Code:**
```typescript
// ❌ The Farms (Individual Files like add.js, sub.js)
class DairyFarm { getMilk() { return "🥛 Milk (f1)"; } }
class ChickenFarm { getEggs() { return "🥚 Eggs (f2)"; } }

// ✅ The Supermarket (utility/index.js Barrel)
class Supermarket {
    static gatherGroceries() {
        return {
            f1: new DairyFarm().getMilk(),
            f2: new ChickenFarm().getEggs()
        };
    }
}

// 🏠 You at Home (file2.js)
// You just visit the Supermarket. You alias f1 back to milk!
const { f1: milk, f2: eggs } = Supermarket.gatherGroceries();
console.log(`Baking a cake with ${milk} and ${eggs}!`);
```
